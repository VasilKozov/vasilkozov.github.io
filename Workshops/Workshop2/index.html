Student workshop: simulation first, real robot second

Basic Robotics Workshop

Use Webots to build a small simulated production cell, then optionally run a safe first program on the Elephant Robotics mechArm 270-Pi.

Workshop outputs

  • ✅ Webots project or modified demo
  • ✅ Robot movement sequence
  • ✅ Production-line or manipulation logic
  • ✅ Short safety explanation
  • ✅ Optional mechArm 270-Pi hardware demo

Normal part

Webots simulation exercise: mini production cell

You will work in Webots and create a small robot automation scenario. The goal is not a perfect factory model. The goal is to show that you understand the robotics workflow: define the workspace, inspect the robot devices, program safe motion, add a simple process rule, and explain what would be different on real hardware.

Main task scenario

Imagine a small production line where parts arrive in a robot work area. A robot must move between safe positions, pick or simulate picking a part, place it in a target area, and optionally assemble a simple composite part.

Level 1

Move the robot through safe predefined positions.

Level 2

Create a repeatable pick-and-place sequence between two zones.

Level 3

Add sorting, inspection, conveyor timing, or a simple assembly idea.

Individual workshop work

Each student completes the exercise independently. You may discuss ideas with colleagues, but your Webots scene, controller logic, screenshots, and short explanation should be your own work.

  • Plan: define your robot task and labelled work zones.
  • Implement: modify the Webots world and controller step by step.
  • Test: run the simulation safely and fix obvious issues.
  • Explain: describe what the robot does and what its limitations are.

Recommended Webots route

Use one of the bundled Webots sample worlds as your base. This is better than starting from an empty world because students can immediately inspect a working controller, devices, robot node, end effector, and scene objects. Good starting points are a Universal Robots industrial-arm scene, an assembly-line/conveyor scene, or any factory environment available in your installed Webots sample browser.

If your Webots installation has a slightly different sample name, keep the task the same: choose a robot-arm or conveyor/factory demo, copy it to your own project folder, and modify the controller and scene.

1

Install and open Webots

Install Webots on your laptop or use a laboratory computer where it is already installed. Open Webots and spend two minutes exploring the interface: scene tree on the left, 3D world in the center, text editor/controller area, and simulation controls at the top.

Scene tree
Find robots, objects, solids, motors, cameras, and sensors.
3D world
Observe collisions, robot motion, and object placement.
Controller

2

Open a sample world and save your own copy

From Webots, open a bundled sample world with an industrial arm, conveyor, assembly-line, or factory environment. Immediately save it into your own project folder so you do not overwrite the original sample.

  1. Open a sample world with a robot arm or factory scene.
  2. Run the simulation once without changing anything.
  3. Observe the default robot behaviour and write down what the controller does.
  4. Save a copy in a folder named robotics_workshop_team_X.
Good practice: do not modify everything at once. First make one small change, run, observe, then continue.
3

Map the production cell

Before programming, define named zones. Use either visible markers in the world or written coordinates. Every person must define at least four zones.

HOME
Safe start and finish pose.
PICKUP
Part appears or waits here.
PROCESS
Inspection or assembly position.
OUTPUT
Finished part is delivered.

Draw a quick sketch or take a screenshot and label these zones. This will make your presentation much clearer.

4

Inspect the robot devices

Open the robot node in the scene tree and identify the devices used by the controller. Typical devices are motors, position sensors, a gripper, a camera, distance sensors, or a supervisor controller.

For a robot arm: list the joint names, gripper/tool name, and any position sensors.
For a mobile robot: list wheel motors, distance sensors, camera, and target object names.

Write these device names into the top of your controller as comments. This helps prevent mistakes when the code becomes longer.

5

Create the first safe movement

Start with movement only. Do not add gripping, collision, sorting, or object manipulation until you see that the robot can move safely between known points.

# Pseudocode: first safe movement
initialize_robot()
set_speed(low)
move_to(HOME)
move_to(PICK_APPROACH)   # above or before the object
move_to(PICK_POINT)      # near the object, but not colliding
move_to(PICK_APPROACH)
move_to(HOME)
Success condition: the robot reaches at least three visible positions without crossing through objects or the table.
6

Add pick-and-place or simulated manipulation

If the sample world already has a gripper or attach/detach behaviour, use it. If not, simulate manipulation by moving to the object, waiting one second, and then moving to the output zone while explaining that this represents a pick action.

# Pseudocode: manipulation sequence
move_to(HOME)
move_to(PICK_APPROACH)
move_to(PICK_POINT)
close_gripper_or_mark_part_as_picked()
move_to(PICK_APPROACH)
move_to(OUTPUT_APPROACH)
move_to(OUTPUT_POINT)
open_gripper_or_mark_part_as_released()
move_to(HOME)

Add short pauses at important points. Pauses make the demonstration easier to understand and make debugging easier.

7

Add production-line logic

Now add a simple decision. You can use a real sensor/camera if the sample has one, or use a variable to represent the part type.

# Pseudocode: production rule
part_type = read_sensor_or_set_variable()

if part_type == "red":
    target = OUTPUT_A
elif part_type == "blue":
    target = OUTPUT_B
else:
    target = REJECT_BIN

pick_part()
place_part(target)
return_home()
Sorting: red/blue parts go to different bins.
Inspection: accepted parts go forward, defective parts are rejected.
Assembly: base part is placed first, cover part second.
8

Make the scenario repeatable

Industrial robot programs are valuable because they are repeatable. Add a loop that performs the task three times, or make three different parts go through the cell.

for cycle in range(3):
    part_type = get_next_part(cycle)
    run_production_cycle(part_type)
    log_result(cycle, part_type)

Keep the loop slow and visible. A workshop demo should be understandable, not fast.

9

Document and present the result

Prepare a short demonstration. You do not need a PowerPoint. Show the Webots world, run the controller, and explain the logic. Include one screenshot or short video if possible.

Minimum explanation

  • What robot is used?
  • What are the zones?
  • What is the movement sequence?
  • What production rule is implemented?
  • What can go wrong in a real factory?

Good engineering reflection

  • Safe approach points are separated from pick/place points.
  • The robot returns home after each cycle.
  • The program is tested in small increments.
  • Limitations are stated honestly.

Advanced part

Real robot exercise: first program on mechArm 270-Pi

The goal is simple and safe: connect to the robot, read its state, move through a few low-speed poses, optionally test the gripper, then return to a safe home pose. This is not a vision task and not an LLM-control task; it is a first reliable hardware-control exercise.

Hardware

  • Elephant Robotics mechArm 270-Pi
  • Raspberry Pi-based controller
  • Official external power supply
  • Optional adaptive gripper
  • Cleared table workspace
  • One instructor operator

Software

  • Python 3
  • pymycobot
  • Terminal or VS Code
  • VNC Viewer or monitor/keyboard/mouse
  • Optional SSH to the Pi

Safety rules

  • Keep hands outside the work envelope.
  • Use low speed first: 15–30.
  • Test without payload first.
  • Stop if the robot moves unexpectedly.
  • Never force a powered joint by hand.
Before coding: boot, power, and connect the robot ⌄
Use the official guide first. Follow Elephant Robotics’ “Unpackaging and First-time Use” and “PI Robot Instructions” pages for the exact boot and connection procedure before students run Python code.
  1. Fix the robot base. Place the mechArm on a stable table/base with enough free workspace around the arm.
  2. Connect the official external power supply. Do not rely on USB Type-C alone for robot-arm power.
  3. Boot the Raspberry Pi system. Wait until the Pi-based system is fully started.
  4. Connect by VNC or direct display. For the Ubuntu 20.04 image, connect to the robot access point ElephantRobotics_AP, password Elephant, then open VNC Viewer to 10.42.0.1. The VNC password is also Elephant. If a monitor is available, connect HDMI, keyboard, and mouse directly.
  5. Open a terminal on the robot. Run python3 --version and confirm the environment is ready.
  6. Keep the robot area clear. The first run should be without payload and with everyone standing outside the motion area.

2

Create a file named hello_mecharm.py

This script tries common ports used in Elephant Robotics / mechArm setups. It starts gently, reads joint angles, moves through safe small poses, and returns to a neutral pose.

from pymycobot import MechArm270
import time

# Common ports for mechArm / myCobot setups.
# On the Pi, /dev/ttyAMA0 or /dev/ttyUSB0 are common.
# On Windows, COM3 is a common example, but your number may differ.
CANDIDATE_PORTS = ["/dev/ttyAMA0", "/dev/ttyUSB0", "/dev/ttyACM0", "COM3"]
CANDIDATE_BAUDS = [1000000, 115200, None]

SAFE_HOME = [0, 0, 0, 0, 0, 0]
POSE_LEFT = [-25, 20, -20, 0, 30, 0]
POSE_RIGHT = [25, 20, -20, 0, 30, 0]
POSE_CENTER = [0, 25, -25, 0, 35, 0]


def connect():
    last_error = None
    for port in CANDIDATE_PORTS:
        for baud in CANDIDATE_BAUDS:
            try:
                print(f"Trying port={port}, baud={baud}")
                arm = MechArm270(port) if baud is None else MechArm270(port, baud)
                time.sleep(1)
                print("Connected. Current angles:", arm.get_angles())
                return arm
            except Exception as exc:
                last_error = exc
    raise RuntimeError(f"Could not connect to mechArm. Last error: {last_error}")


def move(arm, angles, speed=25, wait=3):
    print("Moving to:", angles)
    arm.send_angles(angles, speed)
    time.sleep(wait)
    print("Current angles:", arm.get_angles())


def main():
    arm = connect()

    # Wake / focus servos if the API and firmware support it.
    try:
        arm.power_on()
    except Exception:
        pass

    # First run: no object in gripper, low speed, clear workspace.
    move(arm, SAFE_HOME, speed=20, wait=3)
    move(arm, POSE_LEFT, speed=20, wait=3)
    move(arm, POSE_RIGHT, speed=20, wait=3)
    move(arm, POSE_CENTER, speed=20, wait=3)
    move(arm, SAFE_HOME, speed=20, wait=3)

    print("Demo finished. Releasing servos.")
    arm.release_all_servos()


if __name__ == "__main__":
    main()
3

Run the program under supervision

Clear the robot area, keep the power control accessible, and run the program. Watch how joint-space commands are translated into physical robot movement.

python3 hello_mecharm.py
4

Optional: test the gripper

Only do this after the basic motion script works. If your gripper type is the parallel gripper, your previous project used gripper_type=3. Test with no object first.

# Add after successful connection only, and test at low speed.
# Some gripper APIs differ by firmware/accessory.
try:
    arm.set_gripper_value(100, 30, gripper_type=3)  # open
    time.sleep(2)
    arm.set_gripper_value(1, 30, gripper_type=3)    # close gently
    time.sleep(2)
    arm.set_gripper_value(100, 30, gripper_type=3)  # open again
except TypeError:
    # Fallback for API versions that use set_gripper_state
    arm.set_gripper_state(0, 30)  # open
    time.sleep(2)
    arm.set_gripper_state(1, 30)  # close
    time.sleep(2)
    arm.set_gripper_state(0, 30)  # open
5

Explain what happened

Connect the hardware demo back to the simulation: joint limits, safe approach points, repeatability, payload limits, and why industrial systems use programmed paths instead of improvising motion.

Minimum advanced result: the robot moves through at least three visible positions and safely returns to home.
Troubleshooting checklist ⌄
Cannot connect by VNC
Check that you are connected to the robot access point and that the VNC address is 10.42.0.1.
Import error
Run python3 -m pip install pymycobot --upgrade --user in the same Python environment used to run the script.
Port error
Check whether the robot is controlled through the Pi serial port, USB serial, or a Windows COM port.
No movement
Check external power, firmware state, servo focus, and whether the robot is already in an error state.
Unexpected motion
Stop immediately, reduce speed, simplify poses, and test one joint at a time.
Gripper does not respond
Check accessory type, firmware/API version, and whether the gripper is connected to the correct port.
